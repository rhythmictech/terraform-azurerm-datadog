# terraform-azurerm-datadog

Core Datadog &harr; Azure integration module: it wires the `datadog_integration_azure`
object, an optional per-subscription **Monitoring Reader** role assignment for
the Datadog service principal, and reusable Datadog-provider objects (the `main`
logs index and estimated-usage monitors). It is the Azure sibling of
`terraform-aws-datadog`.

Out of scope (separate modules): the Event Hub + Function **log forwarder** and
the **Defender for Cloud continuous export**. This module leaves clean extension
seams for them but does not implement them.

## Datadog site (US1)

Rhythmic's Datadog org is **US1**, so the `datadog` provider must point at the
US1 API endpoint:

```hcl
provider "datadog" {
  api_key = var.datadog_api_key
  app_key = var.datadog_app_key
  api_url = "https://api.datadoghq.com/" # US1; must NOT end in /api/
}
```

The provider configures the site through **`api_url`** (there is no `site`
argument). A mis-sited org fails loudly against this endpoint. The
`datadog_site_name` variable is retained for parity but does not configure the
provider.

> The Azure Native Datadog (marketplace) resource is US3-only, so this module
> uses the **app-registration** integration (`datadog_integration_azure`), not
> the marketplace resource.

## App-registration boundary (consume vs create)

The default path **consumes** a client-created Datadog app registration: the
client creates the app registration + service principal + client secret and
shares the `client_id`, `tenant_id`, SP **object id**, and `client_secret`. The
module then creates **no** Entra objects, so the deploying identity needs no
Microsoft Graph access. See [`examples/consume`](examples/consume).

Setting `create_app_registration = true` flips to an optional path where the
module creates the `azuread_application` + `service_principal` +
`application_password` itself. That requires a deploying identity with Graph
write (Application Administrator) and is intended for sandbox / self-service
tenants only. See [`examples/create-app-registration`](examples/create-app-registration).

## Noise control via tag filters

Azure has no per-namespace / per-service metric toggles (unlike the AWS
integration's `account_specific_namespace_rules`). Which hosts Datadog pulls in
is controlled entirely by **tag filters**. The convention here is to tag
monitored resources `datadog_managed:true` and set:

```hcl
host_filters = ["datadog_managed:true"]
```

Only resources matching one of the filter tags are imported into Datadog.
`app_service_plan_filters` and `container_app_filters` apply the same idea to
App Service Plans and Container Apps. Empty optional filter lists are sent as
`null` so plans do not churn.

## Role assignment

The Datadog SP's Monitoring Reader assignment is **gated off by default**
(`manage_datadog_sp_role_assignment = false`) because it is commonly pre-created
out-of-band during onboarding by a higher-privilege (Owner) identity, while this
module runs under a lower-privilege identity. Set the flag to `true` only when
this module should own the assignment (and no matching assignment already
exists, else an HTTP 409 `RoleAssignmentExists` results).

## Usage

> **Note:** the registry `source` namespace/name and the `version` pin below are
> provisional and must be confirmed against the first published release tag.

```hcl
module "datadog" {
  source  = "rhythmictech/datadog/azurerm"
  version = "~> 0.1"

  name = "example-client"

  # Consume a client-created Datadog app registration (default path).
  datadog_client_id     = var.datadog_client_id
  datadog_tenant_id     = var.datadog_tenant_id
  datadog_sp_object_id  = var.datadog_sp_object_id
  datadog_client_secret = var.datadog_client_secret

  # Tag-based noise control.
  host_filters = ["datadog_managed:true"]

  # Assignment pre-created during onboarding; scopes still declared so the flag
  # can be flipped later with no further wiring.
  manage_datadog_sp_role_assignment = false
  role_assignment_scopes = [
    "/subscriptions/<sub-guid-a>",
    "/subscriptions/<sub-guid-b>",
  ]
}
```

## Provider notes

The `azuread` provider (`~> 3.0`) is declared unconditionally but is only
exercised when `create_app_registration = true`. On the default (consume) path
all `azuread_*` resources are `count = 0`, so the provider makes no Microsoft
Graph calls and consumers need no Entra/Graph credentials.
